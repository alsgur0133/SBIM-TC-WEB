# 이 프로젝트를 이해하기 — 맥락 가이드

기능 목록이 아니라, **왜 이렇게 짜였는지**와 **코드가 어떻게 이어지는지**를 잡기 위한 문서입니다.

---

## 1. 한 문장으로

**“한 건설/설계 프로젝트를 골라서, 그 프로젝트의 설계 차수·리비전 아래에 도서·모델·물량을 올리고 보고, 필요하면 Trimble Connect와 연결한다.”**

웹(UI)이 보여 주고, **Node 서버가 파일·DB(PostgreSQL)·권한을 처리**합니다. 브라우저만으로는 엑셀 파싱·대용량 업로드를 안전하게 하기 어렵기 때문에 **프론트와 API가 나뉜 구조**입니다.

---

## 2. 두 덩어리: `src/` 와 `server/`

| 덩어리 | 역할 | 비유 |
|--------|------|------|
| **`src/`** (React) | 화면, 버튼, 표, 라우팅, “지금 선택된 프로젝트가 뭐지?” 같은 **UI 상태** | 식당 메뉴판 + 주문 입력기 |
| **`server/index.js`** | 로그인 검증, DB 읽기/쓰기, 파일 저장, 엑셀 파싱, Trimble 토큰 처리 | 주방 + 창고 |

프론트는 **직접 DB 파일을 열지 않습니다.** 전부 `fetch('/api/...')` 같은 호출로 서버에 맡깁니다.

**개발할 때**는 Vite(5173)가 `/api` 요청만 골라서 5001 포트의 Node로 넘깁니다(프록시). 그래서 브라우저 입장에서는 “같은 사이트의 `/api`”로만 말하고, `src/api/config.ts`에서 개발 모드일 때 `API_BASE`를 비워 둡니다.

**배포할 때**는 보통 Node가 `dist` 정적 파일도 같이 서빙하거나, IIS/nginx 앞단에서 같은 호스트로 묶습니다. 이때 경로가 `/SBIM-TC-WEB/` 처럼 **서브경로**면 `vite`의 `base`와 `window.__BASE_PATH__` / `API_BASE` 계산이 맞아야 API가 깨지지 않습니다. 이건 “배포 환경이 URL을 어떻게 쪼개느냐”에 대한 기술적 대응입니다.

---

## 3. 앱이 켜질 때: 라우팅의 맥락

`App.tsx`를 위에서 아래로 읽으면 됩니다.

1. **`AuthProvider`**  
   로그인 여부, 토큰, 사용자 정보가 여기서 관리됩니다. 다른 화면은 `useAuth()`로만 “지금 사용자가 누구냐”를 물어봅니다.

2. **공개 라우트**  
   `/login`, `/signup`, Trimble 콜백 등 — 사이드바 없는 전체 화면.

3. **`/` 아래 메인 앱**  
   `ProjectProvider` + `DesignScheduleProvider`로 감싼 뒤 **`StartOrApp`** 이 들어갑니다.

**`StartOrApp`의 의미**가 중요합니다.

- 로그인 안 됐으면 → `/login`으로 보냄.
- 로그인 됐으면 → **`Layout`(사이드바) + `<Outlet />`(각 페이지)** 을 보여 줌.
- URL에 `?code=&state=` 가 붙은 Trimble OAuth 콜백도 여기서 한 번 처리합니다(PKCE, 서버에 사용자 확인).

즉, “앱의 뼈대”는 **StartOrApp → Layout → 페이지** 입니다.

4. **`ProtectedRoute`**  
   프로젝트 관리, 사용자 관리 등 **역할이 필요한 메뉴**만 감쌉니다. 나머지는 로그인만 되어 있으면 들어갈 수 있습니다(각 페이지 안에서 또 `canManage`로 버튼을 막기도 함).

---

## 4. 세 가지 전역 맥락 — 왜 이렇게 나뉘었나

대부분의 설계/물량 화면은 공통으로 다음을 알아야 합니다.

### 4.1 `ProjectContext` — “지금 어떤 프로젝트를 보고 있나”

- 헤더에서 프로젝트를 고르면 여기 저장됩니다.
- API 많은 곳이 `designRevisionId`만 받더라도, **어느 프로젝트 소속인지**는 리비전/파일이 DB에서 연결되어 있습니다. UI에서는 “프로젝트를 먼저 골라라”는 흐름을 맞춥니다.

### 4.2 `DesignScheduleContext` — “지금 몇 차 설계의 몇 번 리비전인가”

- **설계일정 관리** 화면의 테이블은 **로컬 state**로도 목록을 들고 있습니다(편집 UX 때문).
- **헤더 드롭다운**은 이 **컨텍스트의 `phases` / `revisions`** 를 봅니다.

그래서 예전에는 “설계일정에서 차수만 추가하고 컨텍스트를 안 갱신”하면 헤더가 뒤처지는 **불일치**가 생겼고, 저장/삭제 후 `fetchPhases()`를 컨텍스트에도 호출하도록 맞춘 것입니다.  
**같은 데이터를 두 군데에서 쓰면, 바뀔 때 둘 다 갱신해야 한다**는 맥락입니다.

### 4.3 `AuthContext` — “누구이며 관리자인가”

- 이메일/역할로 `canManage` 등을 켜고 끕니다.
- 서버도 같은 이메일로 “이 사람이 프로젝트 관리자냐”를 다시 검사합니다. **프론트는 편의, 서버가 최종 권한**입니다.

---

## 5. 데이터가 도는 방식(대표 패턴)

1. 사용자가 헤더에서 프로젝트 + 차수 + 리비전 선택.
2. 예: 물량파일 등록 페이지가 `designRevisionId`를 넣어 `GET /api/quantity-files?...` 호출.
3. `server/index.js`가 PostgreSQL에서 읽어 JSON 반환.
4. React가 state에 넣고 표를 그림.

**쓰기**(업로드, 삭제)도 동일하게 POST/DELETE → 서버가 파일 디스크 + DB 트랜잭션.

**물량 엑셀**은 브라우저에서 파싱하지 않고 **서버의 `parseQuantityExcelSheet`** 에 모아 둔 이유는:

- 형식이 여러 종류(산출서, 반입, 집계표)이고 규칙이 길다.
- `xlsx`와 경로·보안을 서버 한곳에서 처리하는 편이 낫다.

---

## 6. 물량 쪽만 조금 더 — 왜 API가 여러 갈래처럼 보이나

- 예전부터 있던 것: **파일 단위** `GET /api/quantity-files/:id/items` 등.
- 대시보드용으로 **리비전 전체를 한 번에** 보기 위해 추가된 것: `GET /api/quantity-revision/stats`, `.../items`.

프론트 `Quantity.tsx`는 통합 API를 쓰되, **구버전 서버**면 404가 나서 파일별 API로 **폴백**하는 코드가 있습니다.  
즉, **“기능은 새로 늘렸는데 서버를 재시작 안 하면 새 경로가 없다”** 같은 운영 맥락이 코드에 반영되어 있습니다.

**B.O.M 보기**는 DB에 “콘크리트 한 줄 + 철근 여러 줄”로 쪼개 저장해 두고, 프론트 `buildBomViewFromItems`가 다시 **엑셀 한 줄처럼** 합쳐 보여 주는 겁니다. 저장 모델과 화면 모델이 다를 때 흔히 쓰는 방식입니다.

---

## 7. Trimble 연동 맥락

- 로그인은 OAuth(PKCE). 토큰·리다이렉트는 **서버 세션/환경 변수**와 **Vite의 `VITE_*`** 가 짝을 맞춰야 합니다.
- “모델 뷰어”는 Connect를 **iframe에 억지로 넣기보다** 별도 창/페이지로 여는 흐름을 택한 부분이 있습니다(제3자 도메인·보안·UX).

자세한 단계는 `TRIMBLE-CONNECT-PROJECT-SYNC.md`가 담당하고, 여기서는 **“인증은 이앱 사용자 DB와 Connect 계정을 엮는 별도 축이다”** 정도만 잡으면 됩니다.

---

## 8. 코드 읽을 때 추천 순서

처음 온 사람이 파일을 열 순서 예시입니다.

1. `src/App.tsx` — 라우트 지도.
2. `src/components/StartOrApp.tsx` — 로그인 후 뭐가 보이는지.
3. `src/components/Layout.tsx` — 메뉴·헤더·프로젝트/차수 선택.
4. `src/contexts/*.tsx` — 전역 상태가 무엇인지.
5. 궁금한 **페이지 하나**만 `src/pages/XXX.tsx` + `src/api/XXX.ts`.
6. 같은 기능의 **`server/index.js`에서 `/api/...` 문자열 검색** — 실제 쿼리와 파일 경로가 나옵니다.

---

## 9. 정리

- **프론트**: 보여 주고, 선택 상태를 들고, API를 호출한다.
- **서버**: 권한·DB·파일·엑셀·외부 OAuth를 처리한다.
- **컨텍스트**: “지금 프로젝트/차수/사용자”를 여러 페이지가 공유한다 — 갱신 타이밍을 놓치면 UI가 어긋난다.
- **배포**: URL 구조(서브경로)와 `API_BASE` / 프록시가 맞아야 “로컬에선 되는데 서버만 안 됨”이 줄어든다.

이 문서는 **의도와 연결 관계**를 잡는 용도이고, 세부 필드·쿼리는 각 파일과 API 응답을 따라가면 됩니다.
